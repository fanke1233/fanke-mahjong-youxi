# 断网重连赖子牌信息丢失问题 - 完整排查记录

**日期**: 2026-05-04  
**项目**: 威海麻将客户端 (xiantaomj.cocos2d_client-main)  
**问题**: 断网重连后赖子牌信息(laiGenTile, laiZiTile)消失  
**状态**: ✅ 已解决

---

## 📋 问题描述

### 初始现象
- 初始发牌后，赖子牌数据正常显示
- 断网重连后，赖子牌信息丢失
- 客户端日志显示：`laiGenTile = undefined, laiZiTile = undefined`

### 影响范围
- 一赖到底玩法中，赖子牌是核心游戏数据
- 赖子牌信息丢失导致胡牌计算、牌型提示等功能异常

---

## 🔍 问题排查过程

### 第一阶段：发现缓存日志残留

#### 现象
虽然之前已删除赖子牌缓存逻辑，但运行时仍输出：
```
[赖子牌缓存] ✅ 已保存赖子牌信息到本地缓存 (laiGenTile=42, laiZiTile=43)
```

#### 根因分析
缓存逻辑隐藏在两个关键文件中：

**1. `__updateMahjongInHand.ts`**
```typescript
// 位置：assets/game/MJ_weihai_/script/table/__updateMahjongInHand.ts
// 问题：手牌更新时将赖子牌保存到本地缓存
if (nLaiGenTile > 0 || nLaiZiTile > 0) {
    let oLaiZiInfo = CachedData.getInstance().get(CachingKeyDef.LAI_ZI_TILE_INFO);
    // ... 保存逻辑 ...
    CachedData.getInstance().set(CachingKeyDef.LAI_ZI_TILE_INFO, oLaiZiInfo);
    cc.log(`[赖子牌缓存] ✅ 已保存赖子牌信息到本地缓存...`);
}
```

**2. `__createMahjongTable.ts`**
```typescript
// 位置：assets/game/MJ_weihai_/script/__createMahjongTable.ts
// 问题：创建牌桌时从本地缓存读取赖子牌
const oCachedLaiZiInfo = CachedData.getInstance().get(CachingKeyDef.LAI_ZI_TILE_INFO);
if (oCachedLaiZiInfo && (oCachedLaiZiInfo.laiGenTile > 0 || ...)) {
    nLaiGenTile = oCachedLaiZiInfo.laiGenTile;
    cc.log(`[赖子牌缓存] ✅ 从本地缓存恢复赖子牌信息...`);
}
```

#### 解决方案
1. 删除 `__updateMahjongInHand.ts` 中的缓存逻辑（约15行代码）
2. 删除 `__createMahjongTable.ts` 中的缓存读取和保存逻辑
3. 删除 `__hintMahjongCanHu` 函数中的缓存读取逻辑
4. 移除 `CachedData` 和 `CachingKeyDef` 的相关导入

### 第二阶段：发现Proto文件版本不一致

#### 现象
清除缓存后，断网重连日志显示：
```
[赖子牌追踪] Player[0] userId = 987346, laiGenTile = undefined, laiZiTile = undefined
[赖子牌追踪] Player[0] 所有字段: mahjongInHand, mahjongOutput, ... (缺少laiGenTile, laiZiTile)
```

#### 根因分析

**服务端确认**：
- ✅ 服务端日志显示正确发送了赖子牌信息
- ✅ Proto消息构建正确：`Player[0] laiGenTile = 48, laiZiTile = 49`

**客户端排查**：

1. **检查Proto定义** ✅
   ```typescript
   // protocol/out/mod_MJ_weihai_Protocol.d.ts
   interface IPlayer {
       laiGenTile?: (number|null);  // 字段编号: 19
       laiZiTile?: (number|null);   // 字段编号: 20
   }
   ```
   Proto定义正确，包含两个字段。

2. **检查Proto解析器** ❌ **发现问题！**
   
   对比两个目录的Proto JS文件：
   
   **最新文件** (`protocol/out/mod_MJ_weihai_Protocol.js`):
   ```javascript
   case 19: {
       message.laiGenTile = reader.sint32();
       break;
   }
   case 20: {
       message.laiZiTile = reader.sint32();
       break;
   }
   ```
   ✅ 包含字段19和20的解码逻辑
   
   **资源目录旧文件** (`assets/game/MJ_weihai_/script/msg/mod_MJ_weihai_Protocol.js`):
   ```javascript
   case 18: {
       message.mahjongLiangFeng = ...;
       break;
   }
   default:  // 直接从case 18跳到default
       reader.skipType(tag & 7);
       break;
   ```
   ❌ **缺少case 19和case 20**，导致解析时跳过这两个字段！

3. **问题根源**
   - 开发者修改了 `.proto` 文件并重新生成了 `protocol/out/` 中的文件
   - **但忘记将新生成的Proto文件同步到资源目录**
   - Cocos Creator运行时加载的是资源目录中的旧文件
   - 客户端使用旧版Proto解析器，无法识别字段19和20

#### 解决方案

**1. 手动同步Proto文件**
```bash
cd "D:\xiantaomj.cocos2d_client-main"
cp protocol/out/mod_MJ_weihai_Protocol.js assets/game/MJ_weihai_/script/msg/
cp protocol/out/mod_MJ_weihai_Protocol.d.ts assets/game/MJ_weihai_/script/msg/
```

**2. 更新自动同步脚本**
在 `protocol/autoGen.cmd` 末尾添加：
```batch
rem # Sync generated files to corresponding msg directories
echo [4/7] Syncing MJ_weihai_Protocol to game/MJ_weihai_/script/msg...
copy /y .\out\mod_MJ_weihai_Protocol.js ..\assets\game\MJ_weihai_\script\msg\mod_MJ_weihai_Protocol.js >nul
copy /y .\out\mod_MJ_weihai_Protocol.d.ts ..\assets\game\MJ_weihai_\script\msg\mod_MJ_weihai_Protocol.d.ts >nul
```

**3. 添加调试日志**
在 `SyncRoomDataResultHandler.ts` 中添加详细追踪：
```typescript
cc.log(`[赖子牌追踪] ====== SyncRoomDataResult 原始数据 ======`);
cc.log(`[赖子牌追踪] oResult.player 是否存在: ${!!oResult.player}`);
cc.log(`[赖子牌追踪] oResult.player 长度: ${oResult.player ? oResult.player.length : 0}`);

for (let i = 0; i < oResult.player.length; i++) {
    const oPlayer = oResult.player[i];
    cc.log(`[赖子牌追踪] Player[${i}] 原始对象:`, oPlayer);
    cc.log(`[赖子牌追踪] Player[${i}] laiGenTile: ${oPlayer.laiGenTile}`);
    cc.log(`[赖子牌追踪] Player[${i}] laiZiTile: ${oPlayer.laiZiTile}`);
    cc.log(`[赖子牌追踪] Player[${i}] 所有字段: ${Object.keys(oPlayer).join(', ')}`);
}
```

## 💡 关键教训

### 1. Proto文件同步至关重要
- ️ 修改 `.proto` 后必须重新生成并**同步到所有资源目录**
- ⚠️ 不能只更新 `protocol/out/`，忘记同步 `assets/` 目录
- ✅ 建议在生成脚本中添加自动同步步骤

### 2. 调试日志的重要性
- 添加详细的追踪日志能快速定位问题
- 日志应包含：字段值、所有字段列表、完整对象JSON
- 使用统一的前缀（如 `[赖子牌追踪]`）便于过滤

### 3. 版本一致性检查
- 遇到问题时，对比 `protocol/out/` 和 `assets/` 目录下的Proto文件
- 使用 `grep` 或 `diff` 工具快速发现差异
- 确认运行时加载的是最新版本

### 4. 避免客户端缓存核心状态
- 赖子牌等关键游戏数据应**完全依赖服务端下发**
- 客户端缓存会导致数据不一致和调试困难
- 遵循"单一数据源"原则

### 5. 断网重连数据完整性
- 服务端在断网重连时必须确保所有关键字段完整
- 客户端应记录详细日志帮助服务端定位问题
- 对比初始发牌和断网重连的数据结构差异
